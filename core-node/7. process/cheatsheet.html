⚙️ Node.js process Module Cheat Sheet

Process control, environment, and runtime introspection — Node.js 18+

💡 Golden Rule: process is a global object — no require/import needed. It gives you access to the runtime environment, CLI arguments, streams, and lifecycle events. Handle its events carefully: unhandled errors and signals can crash or hang your app.
🌍 Environment & Arguments
Env

process.env

process.env

An object containing all environment variables. Read and write (writes only affect current process, not the shell).

// Read
const port = process.env.PORT || 3000;
const nodeEnv = process.env.NODE_ENV || 'development';

// Write (only for current process & children)
process.env.MY_VAR = 'hello';

// Delete
delete process.env.MY_VAR;

// All keys
Object.keys(process.env);
✅ Use dotenv package to load .env files into process.env at startup.
Env

process.argv

process.argv

Array of command-line arguments. Index 0 is the Node.js executable, index 1 is the script path, rest are user arguments.

// node app.js --port 8080 --verbose
process.argv;
// [
//   '/usr/local/bin/node',
//   '/project/app.js',
//   '--port',
//   '8080',
//   '--verbose'
// ]

// Slice user args only
const args = process.argv.slice(2);

// Or use util.parseArgs (Node 18+)
import { parseArgs } from 'node:util';
const { values } = parseArgs({ options: { port: { type: 'string' } } });
Env

process.execPath / execArgv

process.execPath / process.execArgv

Path to the Node.js executable and flags passed to Node.js itself (not your script).

process.execPath;
// → '/usr/local/bin/node'

// node --max-old-space-size=4096 app.js
process.execArgv;
// → ['--max-old-space-size=4096']

process.argv[0] === process.execPath;  // true
📡 Standard I/O Streams
I/O

process.stdin / stdout / stderr

process.stdin / process.stdout / process.stderr

Standard input, output, and error streams. They are streams, so you can pipe, read, and write to them.

// stdin is a Readable stream
process.stdin.on('data', (data) => {
  console.log('You typed:', data.toString().trim());
});

// stdout is a Writable stream
process.stdout.write('Hello\n');  // no newline auto-added

// stderr for errors
process.stderr.write('Error occurred\n');

// Check if stdin is a TTY (interactive terminal)
process.stdin.isTTY;   // true or undefined
process.stdout.isTTY;  // true if outputting to terminal

// Pipe file to stdout
import fs from 'node:fs';
fs.createReadStream('file.txt').pipe(process.stdout);
I/O

process.stdin.setRawMode

process.stdin.setRawMode(mode)

Enable raw mode for reading individual keypresses (no buffering, no echo). Useful for CLI tools.

import readline from 'node:readline';

readline.emitKeypressEvents(process.stdin);
process.stdin.setRawMode(true);

process.stdin.on('keypress', (str, key) => {
  if (key.name === 'c' && key.ctrl) {
    process.exit();  // Ctrl+C
  }
  console.log('Key:', str);
});
⚠️ Only works when stdin.isTTY === true. Will throw in non-TTY environments.
🔄 Lifecycle & Events
Lifecycle

process.on('exit')

process.on('exit', callback)

Fired when the process is about to exit. Synchronous only — no async operations allowed.

process.on('exit', (code) => {
  console.log(`About to exit with code ${code}`);
  // Synchronous cleanup only!
  fs.writeFileSync('/tmp/cleanup.txt', 'done');

  // THIS WILL NOT RUN:
  // setTimeout(() => {}, 0);
  // await anything
});
⚠️ Async code in 'exit' handlers is ignored. Use 'beforeExit' for async cleanup.
Lifecycle

process.on('beforeExit')

process.on('beforeExit', callback)

Fired when the event loop is empty and Node.js is about to exit. Allows async work to keep the process alive.

process.on('beforeExit', async () => {
  console.log('Cleaning up...');
  await closeDatabaseConnection();
  console.log('Cleanup done');
});

// If you add new async work in beforeExit, it can delay exit
// But if process.exit() is called, this event does NOT fire
Lifecycle

process.on('uncaughtException')

process.on('uncaughtException', callback)

Catch synchronous errors that bubble up unhandled. Last resort — log and exit gracefully.

process.on('uncaughtException', (err, origin) => {
  console.error('Uncaught Exception:', err);
  console.error('Origin:', origin);  // 'uncaughtException' or 'unhandledRejection'

  // MUST exit after handling — the process is in an unknown state
  process.exit(1);
});

// Without this handler, the process crashes immediately
throwNewError();  // Would crash without the handler
🔴 Critical: After uncaughtException, the process is in an undefined state. Always log and exit. Never continue running.
Lifecycle

process.on('unhandledRejection')

process.on('unhandledRejection', callback)

Catch Promise rejections that were not handled with .catch(). Future Node.js versions will throw these as uncaught exceptions.

process.on('unhandledRejection', (reason, promise) => {
  console.error('Unhandled Rejection at:', promise);
  console.error('Reason:', reason);

  // Application logic: decide whether to crash or recover
  process.exit(1);
});

// This would trigger the handler:
const p = Promise.reject(new Error('oops'));
// No .catch() attached!
Lifecycle

Signal Events (SIGINT, SIGTERM)

process.on('SIGINT', callback)

Handle OS signals for graceful shutdown. SIGINT = Ctrl+C. SIGTERM = process manager kill (Docker, systemd, Kubernetes).

function gracefulShutdown(signal) {
  console.log(`Received ${signal}. Shutting down...`);
  server.close(() => {
    console.log('HTTP server closed');
    db.disconnect().then(() => {
      console.log('Database disconnected');
      process.exit(0);
    });
  });
}

process.on('SIGINT', () => gracefulShutdown('SIGINT'));
process.on('SIGTERM', () => gracefulShutdown('SIGTERM'));

// Windows doesn't support signals well — use this for cross-platform:
if (process.platform === 'win32') {
  process.on('SIGINT', () => process.exit(0));
}
✅ Always handle SIGTERM in production (Docker, Kubernetes). Default behavior is immediate exit, which drops active connections.
🚪 Exiting & Control
Exit

process.exit

process.exit(code?)

Forcefully terminate the process with an exit code. Skips remaining event loop work and beforeExit event.

process.exit();      // exit code 0 (success)
process.exit(0);     // success
process.exit(1);     // general error

// Common exit codes
// 0   = success
// 1   = uncaught fatal exception
// 3   = internal JavaScript parse error
// 4   = internal JavaScript evaluation failure
// 5   = fatal error
// 6   = non-function internal exception handler
// 7   = internal exception handler run-time failure
// 8   = uncaught exception
// 9   = invalid argument
// 10  = internal JavaScript run-time failure
// 12  = invalid debug argument
// 128 + N = fatal signal (e.g., 130 = SIGINT, 137 = SIGKILL)
⚠️ Warning: process.exit() immediately terminates I/O operations. Close servers and databases first in a graceful shutdown handler.
Exit

process.exitCode

process.exitCode = number

Set the exit code without immediately exiting. The process will exit with this code when the event loop is empty.

process.exitCode = 1;

// Process continues running, will exit with code 1
// when event loop is empty (or on next natural exit)

// Useful for setting error state while finishing work
if (errors.length > 0) {
  process.exitCode = 1;
}
✅ Prefer setting process.exitCode over calling process.exit() when you want to finish pending work.
Exit

process.abort / process.kill

process.abort() / process.kill(pid, signal?)

Hard abort (generates core dump) or send signals to processes.

// Immediate abort — generates core dump, no cleanup
process.abort();

// Send signal to current process
process.kill(process.pid, 'SIGTERM');

// Send signal to another process
process.kill(otherPid, 'SIGINT');

// Check if process exists (signal 0 is a no-op check)
try {
  process.kill(pid, 0);
  console.log('Process exists');
} catch (e) {
  console.log('Process does not exist');
}
ℹ️ Process Info
Info

process.pid / ppid / title

process.pid / process.ppid / process.title

Process ID, parent process ID, and the process name shown in process managers.

process.pid;     // 12345 (current process ID)
process.ppid;    // 12344 (parent process ID)

// Set process title (shown in ps, top, Activity Monitor)
process.title = 'my-api-server';

// Current working directory
process.cwd();   // → '/home/user/project'

// Change working directory
process.chdir('/tmp');
process.cwd();   // → '/tmp'
Info

process.platform / arch / version

process.platform / process.arch / process.version

Runtime and platform information for conditional logic.

process.platform;   // 'darwin' | 'linux' | 'win32' | 'freebsd' | ...
process.arch;         // 'x64' | 'arm64' | 'arm' | 'ia32' | ...
process.version;      // 'v20.11.0' (Node.js version)
process.versions;     // detailed versions object

process.versions.node;     // '20.11.0'
process.versions.v8;       // V8 engine version
process.versions.openssl;  // OpenSSL version
process.versions.zlib;     // zlib version

// Platform-specific logic
if (process.platform === 'win32') {
  // Windows-specific path handling
}
⚡ Performance & Timing
Perf

process.uptime / hrtime

process.uptime() / process.hrtime() / process.hrtime.bigint()

High-resolution timing for benchmarking and measuring intervals.

// Seconds since process started
process.uptime();  // 3600.123 (seconds)

// High-resolution time (nanoseconds, relative, not wall-clock)
const start = process.hrtime();
// ... do work ...
const [seconds, nanoseconds] = process.hrtime(start);
console.log(`Took ${seconds}s ${nanoseconds}ns`);

// BigInt version (easier arithmetic)
const start2 = process.hrtime.bigint();
// ... do work ...
const end2 = process.hrtime.bigint();
console.log(`Took ${Number(end2 - start2) / 1e6}ms`);
✅ Use hrtime for precise benchmarking. It's monotonic (not affected by system clock changes).
Perf

process.memoryUsage / cpuUsage

process.memoryUsage() / process.cpuUsage(previousValue?)

Monitor resource consumption of the current process.

// Memory usage (bytes)
const mem = process.memoryUsage();
mem.rss        // Resident Set Size (total memory allocated)
mem.heapTotal  // V8 heap total size
mem.heapUsed   // V8 heap used
mem.external   // Memory used by C++ objects bound to JS
mem.arrayBuffers // Memory used by ArrayBuffers and SharedArrayBuffers

// CPU usage (microseconds)
const startCPU = process.cpuUsage();
// ... do work ...
const cpu = process.cpuUsage(startCPU);
cpu.user     // CPU time in user mode (microseconds)
cpu.system   // CPU time in kernel mode (microseconds)
Perf

process.nextTick

process.nextTick(callback, ...args)

Schedule a callback to run immediately after the current operation, before any I/O or timers.

console.log('1');
process.nextTick(() => console.log('3'));
Promise.resolve().then(() => console.log('4'));
setTimeout(() => console.log('5'), 0);
console.log('2');

// Output: 1, 2, 3, 4, 5
// Order: sync → nextTick → microtasks (Promise) → timers → I/O → check
⚠️ Recursive nextTick calls can starve the event loop. Use setImmediate for yielding to I/O.
📡 IPC (Inter-Process Communication)
IPC

process.send / process.on('message')

process.send(message, sendHandle?, options?, callback?)

Send messages between parent and child processes when using child_process.fork() or cluster.

// In child process (child.js)
process.on('message', (msg) => {
  console.log('From parent:', msg);
  process.send({ status: 'done', result: 42 });
});

// In parent
import { fork } from 'node:child_process';
const child = fork('./child.js');
child.send({ task: 'heavy-computation' });
child.on('message', (msg) => {
  console.log('From child:', msg);
});
ℹ️ Only available if the process was spawned with an IPC channel (e.g., fork()).
IPC

process.disconnect / connected

process.disconnect() / process.connected

Close the IPC channel and check connection status.

process.connected;   // true if IPC channel is open

process.disconnect();  // Close IPC channel

process.on('disconnect', () => {
  console.log('IPC channel closed');
});
🚀 Pro Pattern — Graceful Shutdown Boilerplate: 
import http from 'node:http';

const server = http.createServer((req, res) => {
  res.end('OK');
}).listen(process.env.PORT || 3000);

function shutdown(signal) {
  console.log(`${signal} received. Starting graceful shutdown...`);

  server.close(async () => {
    console.log('HTTP server closed');
    // await db.disconnect();
    process.exit(0);
  });

  // Force exit after 10s if graceful shutdown hangs
  setTimeout(() => {
    console.error('Forced shutdown after timeout');
    process.exit(1);
  }, 10000);
}

process.on('SIGTERM', () => shutdown('SIGTERM'));
process.on('SIGINT', () => shutdown('SIGINT'));

process.on('uncaughtException', (err) => {
  console.error('Uncaught Exception:', err);
  shutdown('uncaughtException');
});

process.on('unhandledRejection', (reason) => {
  console.error('Unhandled Rejection:', reason);
  shutdown('unhandledRejection');
});

⚠️ Common Pitfalls

MistakeWhy It's WrongFix
Calling process.exit() without cleanupDrops active connections, corrupts dataClose servers/DBs first, then exit
Async code in 'exit' event handlerNever executes — event loop is drainedUse 'beforeExit' or cleanup before calling exit
Not handling SIGTERM in containersDocker kills with SIGKILL after timeout, hard crashAlways handle SIGTERM for graceful shutdown
Continuing after uncaughtExceptionProcess is in undefined state, may corrupt dataLog and process.exit(1)
Recursive process.nextTick()Starves I/O, blocks event loopUse setImmediate() to yield
Modifying process.env and expecting shell to see itOnly affects current process & childrenUse shell export or .env files
