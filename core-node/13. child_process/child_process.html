🔄 Node.js child_process Module Cheat Sheet

Spawn, execute, and manage child processes — Node.js 18+

💡 Golden Rule: Use spawn for streaming large outputs, exec for small outputs, and fork for Node.js subprocesses with IPC. Always use node: prefix. Prefer spawn over exec for security when handling user input.
🚀 Spawning Processes (Streaming)
Spawn

child_process.spawn

spawn(command, args?, options?)

Launch a new process with a command and arguments. Streams stdout/stderr. Best for long-running or large-output processes.

import { spawn } from 'node:child_process';

// Basic spawn
const ls = spawn('ls', ['-la', '/usr']);

ls.stdout.on('data', (data) => {
  console.log(data.toString());
});

ls.stderr.on('data', (data) => {
  console.error(data.toString());
});

ls.on('close', (code) => {
  console.log(`Exited with code ${code}`);
});

// With options (cwd, env, etc.)
const proc = spawn('node', ['script.js'], {
  cwd: '/path/to/project',
  env: { NODE_ENV: 'production' },
  shell: true   // use shell for command parsing
});
✅ Use for: streaming output, long-running processes, large data.
Spawn

spawn options: stdio

stdio: 'pipe' | 'ignore' | 'inherit' | ['pipe', 'pipe', 'pipe']

Control how stdio streams are handled. Useful for piping between processes.

import { spawn } from 'node:child_process';

// Inherit parent's stdio (output goes to console)
const proc1 = spawn('npm', ['install'], {
  stdio: 'inherit'
});

// Pipe stdio (default)
const proc2 = spawn('grep', ['pattern'], {
  stdio: ['pipe', 'pipe', 'pipe']
});

// Pipe to another process (Unix style)
const ls = spawn('ls', ['-la']);
const grep = spawn('grep', ['js']);
ls.stdout.pipe(grep.stdin);
grep.stdout.on('data', data => console.log(data.toString()));
✅ Use inherit for CLI tools. Use pipe for programmatic access.
📦 Executing Commands (Buffered)
Exec

child_process.exec

exec(command, options?, callback?)

Execute a command in a shell and buffer the output. Returns all output at once. Good for small outputs.

import { exec } from 'node:child_process';

// With callback
exec('ls -la /usr', (error, stdout, stderr) => {
  if (error) {
    console.error(`Error: ${error.message}`);
    return;
  }
  console.log(stdout);
  console.error(stderr);
});

// With options (timeout, maxBuffer)
exec('npm install', {
  cwd: '/project/path',
  timeout: 30000,        // 30 seconds
  maxBuffer: 1024 * 1024 // 1 MB max output
}, (error, stdout) => {
  // ...
});

// Promise wrapper (util.promisify)
import { promisify } from 'node:util';
const execAsync = promisify(exec);
const { stdout, stderr } = await execAsync('ls -la');
⚠️ Uses shell by default. Security risk with user input. Uses /bin/sh on Unix, cmd.exe on Windows.
Exec

child_process.execFile

execFile(file, args?, options?, callback?)

Execute a file directly without a shell. More secure than exec and slightly faster.

import { execFile } from 'node:child_process';

// Execute binary directly
execFile('node', ['-e', 'console.log("Hello")'], (err, stdout) => {
  console.log(stdout); // 'Hello\n'
});

// With full path
execFile('/usr/bin/curl', ['https://api.ipify.org'], (err, stdout) => {
  console.log(stdout); // IP address
});

// Promise version
import { promisify } from 'node:util';
const execFileAsync = promisify(execFile);
const { stdout } = await execFileAsync('node', ['--version']);
✅ More secure than exec. Recommended when you don't need shell features.
🔄 Node.js Subprocesses (IPC)
Fork

child_process.fork

fork(modulePath, args?, options?)

Spawn a new Node.js process with an IPC channel for message passing. Great for CPU-intensive tasks.

import { fork } from 'node:child_process';

// Parent process
const child = fork('./worker.js', ['arg1', 'arg2']);

// Send message to child
child.send({ type: 'task', data: 'Hello' });

// Receive message from child
child.on('message', (message) => {
  console.log('From child:', message);
});

child.on('exit', (code) => {
  console.log(`Child exited with code ${code}`);
});

// worker.js (child process)
// process.on('message', (msg) => {
//   console.log('From parent:', msg);
//   process.send({ result: 'OK' });
// });
✅ Use for: CPU-intensive tasks, parallel processing, worker pools. IPC is serialized JSON.
Fork

fork options: silent, execArgv

fork(module, args, { silent, execArgv, ... })

Control stdio and Node.js arguments for forked processes.

import { fork } from 'node:child_process';

// Silent: prevents child from inheriting parent's stdio
const child1 = fork('./worker.js', [], {
  silent: true
});
// child1.stdout/stderr are not connected to parent

// Pass Node.js arguments to child
const child2 = fork('./worker.js', [], {
  execArgv: ['--inspect', '--max-old-space-size=4096']
});

// Custom environment
const child3 = fork('./worker.js', [], {
  env: { NODE_ENV: 'production', CUSTOM_VAR: 'value' }
});
ℹ️ execArgv lets you pass V8 flags to child processes.
🔧 Utility Functions
Util

child_process.spawnSync / execSync / execFileSync / forkSync

spawnSync(command, args?, options?) / execSync(command, options?)

Synchronous versions of spawn/exec/execFile. Block the event loop.

import { spawnSync, execSync, execFileSync } from 'node:child_process';

// Synchronous spawn
const result = spawnSync('ls', ['-la']);
console.log(result.stdout.toString());
console.log(`Status: ${result.status}`);

// Synchronous exec
const output = execSync('echo "Hello World"');
console.log(output.toString()); // 'Hello World\n'

// Synchronous execFile
const nodeVersion = execFileSync('node', ['--version']);
console.log(nodeVersion.toString());

// ⚠️ These block the event loop!
// Only use during startup or in CLI scripts.
⚠️ Synchronous methods block the event loop. Only use during server startup or in CLI scripts.
Util

Process signals & cleanup

child.kill(signal?) / child.on('error') / child.on('exit')

Send signals, handle errors, and clean up child processes.

import { spawn } from 'node:child_process';

const child = spawn('node', ['server.js']);

// Handle errors
child.on('error', (err) => {
  console.error('Process error:', err);
});

// Handle exit
child.on('exit', (code, signal) => {
  console.log(`Exited with code ${code}, signal ${signal}`);
});

// Kill with signal
child.kill();          // SIGTERM
child.kill('SIGKILL'); // Force kill
child.kill('SIGINT');  // Ctrl+C

// Cleanup on parent exit
process.on('exit', () => {
  if (!child.killed) {
    child.kill();
  }
});
✅ Always handle error and exit events. Clean up child processes on parent exit.
🚀 Pro Pattern — Safe Command Execution: 
import { spawn } from 'node:child_process';

// Safe: arguments are passed as array, no shell
function safeExec(cmd, args, options = {}) {
  return new Promise((resolve, reject) => {
    const proc = spawn(cmd, args, {
      ...options,
      shell: false // Security: no shell injection
    });
    
    let stdout = '';
    let stderr = '';
    
    proc.stdout.on('data', data => stdout += data);
    proc.stderr.on('data', data => stderr += data);
    
    proc.on('close', (code) => {
      if (code !== 0) {
        reject(new Error(`Exit code: ${code}\n${stderr}`));
      } else {
        resolve({ stdout, stderr });
      }
    });
  });
}

// Usage
const { stdout } = await safeExec('ls', ['-la'], { cwd: '/usr' });

📋 Quick Decision Guide

GoalUseWhy
Streaming large outputspawn()Event-driven, memory-efficient
Small output, simple commandexec()Easy, buffers output
No shell, more secureexecFile()Direct execution, no shell
Node.js subprocess with IPCfork()Message passing, CPU tasks
Synchronous (CLI/startup)spawnSync() / execSync()Blocks event loop, simple
Piping between commandsspawn() with stdio: 'pipe'Unix-style piping

⚠️ Common Pitfalls

MistakeWhy It's WrongFix
Using exec() with user inputShell injection vulnerabilityUse spawn() with args array or execFile()
Not handling error eventProcess errors can crash your appAlways attach .on('error') handler
Forgetting to kill child processesZombie processes, resource leaksKill on parent exit or timeout
Using execSync() in serverBlocks event loop, hurts performanceUse async versions or spawn
Not setting maxBuffer for exec()Buffer overflow for large outputsSet appropriate maxBuffer or use spawn()
Assuming fork() works on WindowsRequires Node.js, works cross-platformIt does work, but uses node executable
